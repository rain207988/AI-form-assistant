# Chat2Excel 销售报表 RAG 优化说明

## 1. 这次为什么要改

这轮不是做“通用知识库增强”，而是把当前 RAG 明确收敛到 **销售报表** 场景。

销售部门日常问题通常集中在这些方向：

- 查某月/某季度/某区域销售额
- 看客户、产品、业务员排行
- 对比销量、毛利、GMV、营收等指标
- 按区域、渠道、客户、产品做汇总分析
- 对部分销售数据做修正后继续追问

原本那版 RAG 已经能用，但更偏“文件级通用检索”。对于销售报表，有几个明显短板：

1. 只靠向量相似度，`销售额 / 成交额 / GMV / 营收` 这类词不够稳
2. 多 sheet 时容易被某一个高分 chunk 带偏，选错表
3. 样例数据只取前几行，时间排序或区域排序的报表很容易失真
4. 修改数据后，RAG 缓存不会立刻失效，后续追问可能读到旧上下文

所以这次优化的目标很明确：

- 让销售术语识别更稳
- 让多表选表更准
- 让样例数据更能代表整张销售表
- 让修改后的追问不再使用旧索引

## 2. 改了哪些文件

### 2.1 `ai-service/src/main/java/com/bitejiuyeke/ai/config/RagProperties.java`

新增了销售报表专用的 RAG 配置项：

- `maxMatchedTables`
  - 表级召回最多保留多少张候选表
- `semanticWeight`
  - 语义向量得分权重
- `lexicalWeight`
  - 关键词命中得分权重
- `tableScoreThreshold`
  - 表级召回阈值
- `exactMatchBoost`
  - 表头/业务词精确命中时的加分
- `salesSynonymGroups`
  - 销售术语别名组

### 2.2 `ai-service/src/main/java/com/bitejiuyeke/ai/dto/rag/RagContext.java`

新增：

- `rankedTables`
  - 保存表级排序结果，例如 `sales_q1(0.8732)`

作用：

- 让前端或主流程可以看到“这次 RAG 认为哪些表最像”
- 后续如果你想做调试面板或日志分析，会更直观

### 2.3 `ai-service/src/main/java/com/bitejiuyeke/ai/service/RagService.java`

新增：

- `invalidate(Long fileId)`

作用：

- 在销售数据被修改后，主动清理当前文件的 RAG 缓存
- 防止“刚改完数据，接着问还是旧上下文”

### 2.4 `ai-service/src/main/java/com/bitejiuyeke/ai/service/impl/RagServiceImpl.java`

这是这次最核心的改动，已经从“轻量文件级通用 RAG”增强成了“销售报表定向 RAG”。

### 2.5 `ai-service/src/main/java/com/bitejiuyeke/ai/service/impl/AiServiceImpl.java`

改了两件关键事情：

1. SSE 检索摘要里增加候选表排序信息
2. 修改 SQL 执行成功后，立即调用 `ragService.invalidate(fileId)`

### 2.6 `ai-service/src/main/java/com/bitejiuyeke/ai/service/impl/AiModelServiceImpl.java`

强化了提示词，使 SQL 生成和字段提取更贴近销售报表语义。

### 2.7 `ai-service/src/main/resources/bootstrap.yml`

增加了销售 RAG 的默认配置：

- 术语别名组
- 表级召回参数
- 混合召回权重

## 3. RAG 现在具体增强了什么

## 3.1 从“纯向量召回”升级为“混合召回”

以前：

- 只对用户问题做 embedding
- 只按 cosine similarity 排序 chunk

现在：

- 先识别用户问题里的销售业务词
- 再做术语归一化和别名扩展
- 同时计算：
  - 语义向量得分
  - 关键词命中得分
- 最后按加权综合分排序

### 这有什么用

例如用户问：

- “查一下华东地区本月 GMV”

现在系统更容易把这些词对齐到：

- `GMV -> 销售额 / 成交金额 / 营收`
- `华东 -> 区域`
- `本月 -> 日期 / 月份`

这对销售报表比纯向量召回更稳。

---

## 3.2 增加了销售术语别名字典

默认内置了这些别名组：

- `销售额, 成交额, 成交金额, GMV, 营收, 收入`
- `销量, 销售量, 件数, 出货量, 销售件数`
- `毛利, 利润, 利润额, 毛利润`
- `客户, 商户, 经销商, 代理商, 门店`
- `区域, 大区, 省区, 片区, 城市, 地区`
- `产品, 商品, 品类, SKU, 型号, 品牌`
- `渠道, 渠道类型, 销售渠道, 销售通路`
- `销售员, 业务员, 客户经理, 招商专员`
- `日期, 时间, 月份, 月度, 季度, 年度`

### 这有什么用

如果你们部门不同同事说法不统一，比如：

- 有人说“GMV”
- 有人说“销售额”
- 有人说“成交金额”

现在 RAG 会先把它们归一到一组业务概念，再去选表和拼上下文。

---

## 3.3 从“chunk 第一名选表”升级为“表级聚合选表”

以前：

- 哪个 chunk 分高，就优先信哪个 chunk 对应的表

现在：

- 先对 chunk 打分
- 再把同一张表下多个 chunk 的得分聚合起来
- 形成表级排序
- 再把前几名表作为候选表返回

### 这有什么用

如果一个销售文件里有多个 sheet：

- `按月销售`
- `客户回款`
- `区域汇总`
- `业务员业绩`

以前某个样例数据片段偶然命中高分，就可能误选。

现在会综合考虑：

- 这张表命中了多少业务词
- 命中的 chunk 是否稳定
- 是否命中了归一化后的销售术语

这样多 sheet 的表选择会更稳。

---

## 3.4 样例数据采样从“只取前几行”升级为“更有代表性”

以前：

- 直接 `limit N`
- 只取表头部前 N 行

现在：

- 取头部样例
- 取尾部样例
- 再补一部分随机样例
- 合并去重后作为最终采样集

### 这有什么用

销售报表很常见的情况是：

- 按日期排序
- 按地区排序
- 按客户排序

如果永远只看前 15 行，RAG 看到的往往只是：

- 某一个月
- 某一个区域
- 某一类客户

现在样例覆盖会更广，更有利于：

- 时间范围识别
- 区域字段理解
- 客户/产品长尾值感知

---

## 3.5 给每张表补了“业务维度摘要”和“时间维度摘要”

新的表级 chunk 不只包含：

- 文件名
- sheet 名
- 表名
- 行数
- 字段映射

还会尝试补充：

- 哪些字段像维度字段
- 哪些字段像指标字段
- 哪些字段像时间字段
- 时间字段覆盖了哪些月份/日期
- 某些维度字段的样例值

### 这有什么用

销售报表里最常问的是：

- 某个月销售额
- 某区域销量
- 某客户回款
- 某产品毛利趋势

这些问题对“时间字段”和“维度字段”的识别要求很高。

这次补充的摘要能帮助模型更快知道：

- 哪列像日期
- 哪列像区域
- 哪列像客户
- 哪列像数值指标

---

## 3.6 Prompt Context 变得更结构化

以前 RAG 只是把命中的 chunk 文本拼进去。

现在 prompt 里会明确给出：

- 优先候选表
- 表级得分排序
- 用户业务词归一化结果
- 每个片段的：
  - 表名
  - 类型
  - 综合得分
  - 语义得分
  - 关键词得分

### 这有什么用

模型在生成 SQL 时，不再只是“看到一堆文本”，而是能知道：

- 当前最像的是哪张销售表
- 为什么像
- 命中的业务词是什么

这对销售报表里的多义词、近义词和多表场景特别重要。

---

## 3.7 修改数据后，RAG 缓存立即失效

以前：

- 执行完 `update` 后
- RAG 缓存还会继续保留到过期时间

现在：

- 修改 SQL 执行成功后
- 立即 `invalidate(fileId)`

### 这有什么用

销售运营场景里常见连续动作是：

1. “把华南区 3 月销售额修正一下”
2. “再帮我看一下修正后的区域销售排行”

如果缓存不失效，第二问可能还是旧上下文。

现在这类连续操作会更安全。

---

## 3.8 字段提取和 SQL 提示词更贴销售场景

在 `AiModelServiceImpl` 里补了针对销售报表的提示：

- 字段提取优先识别：
  - 销售额
  - 销量
  - 毛利
  - 客户
  - 区域
  - 产品
  - 时间
- 查询 SQL 更明确要求：
  - 理解销售业务词
  - 处理排行、汇总、趋势、对比
  - 更积极使用 `group by / sum / avg / order by`
- 修改 SQL 更强调：
  - where 条件尽量精确
  - 避免误改整张销售表

### 这有什么用

这一步不是检索本身，但它能让 RAG 召回到的上下文更好地传递给大模型。

更适合的问题包括：

## 5. 这轮优化的实际业务价值

如果你的目标是“为部门解决销售报表问题”，这轮改动最直接的价值是：

1. 降低多 sheet 选错表的概率
2. 提升销售类同义词理解能力
3. 提升时间、区域、客户、产品这类分析维度的识别能力
4. 减少修改后继续追问时的脏缓存问题
5. 让模型更容易生成销售分析常见的聚合 SQL

## 6. 当前还没做，但后面值得继续做的点

这轮已经够你先拿去做销售场景验证，但如果后面要继续升级，我建议按这个顺序往下走：

### 6.1 做“可配置的部门术语词典”

现在术语别名是默认内置的。

后续可以把它做成：

- Nacos 配置
- 数据库配置
- 或前端管理页

这样你们部门自己就能维护：

- 特有渠道名
- 产品线叫法
- 业务口径别名

### 6.2 做“时间表达规范化”

例如把：

- 本月
- 上月
- 本季度
- 去年同期
- 环比
- 同比

预先解析成明确时间条件，再喂给 SQL 生成，会更稳。

### 6.3 做“字段级 rerank”

现在已经有表级聚合，但如果后面你们报表字段更多、更复杂，可以增加：

- 表级选择后
- 再做字段级重排

这样 SQL 生成会更准。

### 6.4 做“查询结果口径模板”

比如销售部门喜欢的回答结构固定成：

- 结论
- 核心指标
- 变化趋势
- 异常提醒

这样最终 AI 回答会更像业务助手，而不是普通聊天模型。
